ASSESS-001

Executive Summary (revised)

Proceed

v1 conflated two very different ideas. Migrating SoT/EPIC source files from markdown to HTML is still a bad trade (token cost, LLM authoring fragility, toolchain rewrite). But the approach you’re actually proposing — markdown source rendered through methodology-owned HTML templates, with visible render-date frontmatter — is materially different and materially better.

In the rendering approach: Claude still reads markdown (zero token regression), every script and skill keeps working unchanged, and you gain a presentation surface where design-heavy SoT files (DESIGN_COMPONENTS, architecture, screen flows) become genuinely interactive — not just prettier.

Refined Recommendation Ship HTML templates as first-class methodology artifacts alongside markdown templates. Build a small generator. Publish output via GitHub Pages. Treat the rendered HTML as the consumption surface for HeartBeat. Keep markdown as canonical source forever.

DESIGN-001

Design-Content Unlocks (what v1 missed)

Major Gain

v1 measured the format trade purely in text-content terms (entry tokens, regex parsing). That undersells what HTML enables for design-heavy SoT files. The methodology has at least four files where the canonical content is not prose — it’s visual artifact. Markdown can only describe these; rendered HTML can be them.

Where rendering transforms the experience

SoT.DESIGN_COMPONENTS.md (DES-) Hex codes in a table, font name as text Live color swatches, type specimens, button states, component previews
SoT.TECHNICAL_DECISIONS.md (ARC-) ASCII diagrams or linked PNGs that go stale Inline Mermaid / SVG architecture diagrams, clickable to drill into IDs
SoT.USER_JOURNEYS.md (SCR-, UJ-) Bullet list of screens Wireframe boxes, journey flow diagrams, step-by-step screen previews
SoT.DATA_MODEL.md (DBT-) Markdown tables of columns Rendered ER diagrams, relationship visualizations
SoT.API_CONTRACTS.md (API-) JSON code blocks Swagger-style endpoint browsers, expandable schemas

RENDER-001

Render-Date Frontmatter Pattern

Adopt as Standard

Your concern about source-vs-output discrepancy is real and exactly the right one to surface in the UI. The pattern at the top of this document — the visible Last rendered / Source / Freshness block — should be part of every rendered SoT and EPIC.

Recommended fields

FieldPurposeSource
last_rendered When the HTML was generated Generator timestamp at render time
source_path Which markdown produced this HTML Generator input
source_last_modified When the markdown last changed git log -1 --format=%cI <file>
generator_version Which version of the renderer produced this Pinned in render-sot-html.py
template Which template was used Generator config
freshness Visual indicator: current / stale / expired Computed: source_last_modified vs last_rendered

Freshness rules (proposed)

  • Current — render is newer than source. Trustworthy.
  • Stale — source modified after last render, < 24h ago. Reader knows a re-render is pending.
  • Expired — source modified > 24h before render check. Reader should treat this view as potentially wrong; CI should have re-rendered.
Why visible, not hidden Hidden metadata only helps tooling. Visible frontmatter teaches the reader to ask “is this view current?” before acting on it — the same discipline SoT IDs encode for structure.

PAGES-001

GitHub Pages, Explained

Reference

What it is

GitHub Pages is free static-site hosting built into every GitHub repository. You point it at a branch (typically gh-pages) or a folder (typically /docs on main), and on every push GitHub serves the HTML/CSS/JS in that location as a real website at: 

https://mattgierhart.github.io/PRD-driven-context-engineering/

What “Render SoT to GitHub Pages” means concretely

  1. Generator runs locally or in CI: reads markdown SoT/EPIC files, applies templates, writes HTML.
  2. Output lands in docs/ (or commits to gh-pages branch).
  3. A GitHub Actions workflow triggers on push to main; re-renders; publishes.
  4. The rendered files are accessible at the Pages URL by anyone with the link.
  5. HeartBeat (or any other consumer) can fetch these URLs without cloning the repo.

What you get

  • A real URL for every SoT and EPIC document, with full styling
  • Free hosting, scale handled by GitHub’s CDN
  • HTTPS, custom domain support (optional)
  • Browseable index of all rendered files
  • The render-date frontmatter is publicly verifiable — readers see freshness without cloning

Minimum viable setup

# .github/workflows/render-sot.yml
name: Render SoT to GitHub Pages
on:
  push:
    branches: [main]
    paths: ['SoT/**', 'epics/**', 'templates/**', 'scripts/render-sot-html.py']

jobs:
  render:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install jinja2 markdown beautifulsoup4 pyyaml
      - run: python scripts/render-sot-html.py --out docs/
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./docs

Settings → Pages → Source: gh-pages branch, / (root). Done.

HB-001

HeartBeat Integration Surface

Assumes Context

Note: HeartBeat isn’t in the methodology files yet — only the v0.7 “core loop” uses the term. The analysis below assumes HeartBeat is a companion product that exposes project health/readiness/structure to a non-developer audience (stakeholders, designers, ops). Correct this framing if it’s off and I’ll re-tune.

Under that assumption, the rendered HTML output is the natural integration surface. HeartBeat doesn’t need to parse markdown or run the readiness scorer locally — it consumes the published HTML + the JSON it sits alongside.

What HeartBeat would consume

ArtifactFormConsumer pattern
Rendered SoT files HTML on GitHub Pages Embed via iframe, fetch and re-style, or extract structured data attributes
Readiness scores status/readiness.json Already machine-readable; render badges, trend charts
Render-date frontmatter HTML metadata block (visible) + JSON sibling (machine) Show freshness state in HeartBeat dashboard
EPIC progress Rendered EPIC HTML + frontmatter Pull session state, phase completion, observation count

What it implies for the renderer

  • Emit a JSON sibling for every rendered HTML file (SoT.BUSINESS_RULES.html + SoT.BUSINESS_RULES.json) — gives HeartBeat structured access without HTML parsing.
  • Use stable HTML attributes (data-id, data-status, data-readiness) so HeartBeat can extract entries from the DOM if it chooses to consume HTML directly.
  • Publish a manifest (docs/index.json) listing every rendered file with last-render times. HeartBeat polls this; doesn’t need to crawl.
  • Treat data-id as the canonical ID anchor — HeartBeat can deep-link to a specific BR/UJ/ARC by URL fragment.
Strategic framing If HeartBeat is the companion product, this rendering pipeline isn’t just a presentation tool — it’s the contract between methodology and product. That promotes the templates and the JSON sibling format from “nice-to-have” to API surface. Worth treating as such.

TPL-001

HTML Templates as First-Class Methodology

Worth Doing

v1 framed HTML templates as a downside (“every template needs a twin”). In your refined direction, templates become a methodology asset — shipped, versioned, reusable across projects that adopt this methodology.

Proposed template structure

templates/
├── base.html.jinja                  # Shared layout: header, TOC, footer
├── sot-default.html.jinja           # Generic SoT entry rendering
├── sot-design.html.jinja            # DES- entries: swatch grids, type specimens
├── sot-architecture.html.jinja      # ARC- entries: mermaid/SVG diagrams
├── sot-screen-flow.html.jinja       # SCR-/UJ- entries: wireframes, journey flows
├── sot-data-model.html.jinja        # DBT- entries: ER diagrams
├── sot-api.html.jinja               # API- entries: endpoint browser
├── epic.html.jinja                  # EPIC progress + session state
└── partials/
    ├── render-frontmatter.html.jinja
    ├── status-pill.html.jinja
    ├── id-chip.html.jinja
    └── readiness-badge.html.jinja

Per-prefix template selection

Generator picks a template by ID prefix — DES- entries route through sot-design.html.jinja with swatch and component partials; ARC- entries route through sot-architecture.html.jinja with a Mermaid renderer; etc. The default template handles text-heavy entries (BR, UJ, API spec).

Author ergonomics stay markdown-native

Authors edit markdown files as they do today. To add rendering hints (color values, diagram source, wireframe boxes), they use fenced code blocks with language tags the generator recognizes: 

### DES-005: Status Color Tokens

```color-tokens
accent: #2563eb
pass:   #16a34a
warn:   #d97706
block:  #dc2626
```

Used by readiness badges and status pills across all rendered views.

The markdown is still readable as-is in GitHub. The renderer turns the color-tokens block into the swatch grid you saw in DESIGN-001. No HTML in the source, ever.

CAV-001

Honest Caveats

Read Before Adopting

1. Drift discipline lives in CI, not in vibes

The render-date frontmatter shows staleness; CI has to fix it. Without an auto-render-on-push workflow, you’ll start trusting outdated HTML and not notice. The freshness pill is a warning system, not a solution — the solution is “source change ⇒ render runs before merge.”

2. Design-heavy templates can become their own project

A solid sot-architecture.html.jinja with proper Mermaid rendering, dark mode, and printable export takes more than the half-day budgeted. Start with one or two prefixes; expand opportunistically.

3. Rendering hides errors the markdown didn’t have

If the generator silently skips an unrecognized code-fence language, authors won’t notice until someone views the page. Build a --strict mode that fails on unknown blocks; run it in CI.

4. The JSON sibling format is an API once published

If HeartBeat starts consuming docs/*.json, you can’t change the schema casually. Version it from day one ({ "$schema_version": "1.0" }) so future changes have a path.

5. Visible render-date adds a discipline tax

Users will see “Last rendered: 12 days ago” and ask why. That’s a feature, but it costs attention. Make sure the CI workflow is actually reliable before publishing the freshness indicator publicly — otherwise it just looks broken.

REC-001

Final Recommendation (v2)

Proceed
  1. Build the generator + template system as a methodology asset — not a side script. It deserves its own SoT section (SoT.METHODOLOGY.md or extension of SoT.README.md) describing the contract, the templates, the freshness rules.
  2. Adopt the visible render-date frontmatter from day one. The discrepancy concern you raised is exactly the right one; make the answer part of the methodology, not an afterthought.
  3. Publish via GitHub Pages. Free, public, real URLs, CDN. Adds the “non-developer can read this” tier the methodology has been missing.
  4. Emit a JSON sibling and a manifest for HeartBeat. Treat them as API surface; version-tag them. This is the integration contract whether HeartBeat is built next week or next year.
  5. Start with design-heavy prefixes (DES, ARC, SCR). They have the highest gain-to-effort ratio; text-heavy ones (BR, API) can keep the default template until you have specific reasons to specialize.

What changed from v1: v1’s recommendation against migration still stands — if you meant “HTML source replacing markdown.” What you actually meant (and what your instincts about HeartBeat point to) is rendering, not migration. Those are different decisions and only one of them is costly.
