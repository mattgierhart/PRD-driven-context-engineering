CTX-001

Context & Goal

Why

SoT and EPIC files are authored in markdown for excellent reasons: low token cost when read by Claude, clean diffs, easy hand-editing, native GitHub preview. That stays.

What markdown does not serve well is the design-heavy content carried by certain SoT files — component swatches, architecture diagrams, screen-flow wireframes, ER diagrams, API endpoint browsers. For those, the canonical content is visual, and markdown can only describe it.

Goal: build a small generator and a set of HTML templates that render the existing markdown SoT/EPIC files into styled HTML, publish the output via GitHub Pages, and document the pipeline in SoT/SoT.README.md so it becomes a first-class methodology asset.

Markdown remains the source of truth. HTML is generated output. Authors never edit HTML by hand. Every rendered file shows a visible last rendered indicator so readers can tell whether they are looking at a current view.

SCOPE-001

Scope & Out of Scope

Boundaries

In Scope

  • Python generator script (scripts/render-sot-html.py)
  • Jinja2 templates: one base, one default SoT, three specialized (design, architecture, screen-flow), one EPIC
  • GitHub Actions workflow that publishes to GitHub Pages on push to main
  • JSON companion file per rendered HTML + manifest at docs/index.json
  • Documentation section in SoT/SoT.README.md
  • Render-date frontmatter pattern visible on every rendered page

Out of Scope

  • Changing markdown source files except to add code-fence blocks for design-heavy entries
  • Rewriting existing scripts, hooks, or skills (none of them change)
  • Replacing markdown templates (EPIC_TEMPLATE.md, etc.) — they stay
  • Client-side search, filtering, dark mode, print stylesheets — future iterations
  • Specialized templates for DBT- and API- prefixes — ship default template for now
  • Authentication or access control on the published Pages site

STEP-001

Step 1 · Generator MVP

~1 day
Goal:
Render a single SoT markdown file into styled HTML using the base template.
Inputs:
Any existing SoT/*.md file (start with SoT.UNIQUE_ID_SYSTEM.md)
Outputs:
docs/SoT/SoT.UNIQUE_ID_SYSTEM.html

Procedure

  1. Add dependencies to requirements.txt (or pyproject.toml): jinja2 >= 3.1, markdown >= 3.5, pymdown-extensions >= 10.0, pyyaml >= 6.0, python-frontmatter >= 1.0.
  2. Create scripts/render-sot-html.py with a CLI: render-sot-html.py <file|all> [--out docs/] [--strict].
  3. Implement scripts/_render/parser.py: load file, split YAML frontmatter, render markdown body to HTML, extract entries by H2/H3 ID heading using the same HEADING_DEF_RE pattern as scripts/_readiness/common.py (consistency).
  4. Create templates/base.html.jinja with the layout shown in this document (header, sticky TOC, footer). All CSS inline in the template.
  5. Create templates/sot-default.html.jinja extending base. Renders the entries list with ID-anchored sections.
  6. Create templates/partials/render-frontmatter.html.jinja — see SPEC-001.
  7. Generator computes render-date metadata: invocation time, git log -1 --format=%cI <file> for source mtime, version pinned in script.
  8. Write output. Open in browser. Verify.

Starter script skeleton

#!/usr/bin/env python3
"""Render SoT/EPIC markdown to styled HTML.

Usage:
  render-sot-html.py <file|all> [--out docs/] [--strict]
"""
from __future__ import annotations
import argparse, json, subprocess, sys
from datetime import datetime, timezone
from pathlib import Path

import frontmatter, markdown
from jinja2 import Environment, FileSystemLoader, select_autoescape

VERSION = "0.1.0"
REPO_ROOT = Path(__file__).resolve().parent.parent
TEMPLATES = REPO_ROOT / "templates"

PREFIX_TEMPLATE = {
    "DES": "sot-design.html.jinja",
    "ARC": "sot-architecture.html.jinja",
    "SCR": "sot-screen-flow.html.jinja",
    "UJ":  "sot-screen-flow.html.jinja",
    "EPIC": "epic.html.jinja",
}
DEFAULT_TEMPLATE = "sot-default.html.jinja"

def git_mtime(path: Path) -> str:
    out = subprocess.check_output(
        ["git", "log", "-1", "--format=%cI", "--", str(path)],
        cwd=REPO_ROOT, text=True,
    ).strip()
    return out or datetime.now(timezone.utc).isoformat()

def freshness(source_iso: str, render_iso: str) -> str:
    src = datetime.fromisoformat(source_iso)
    rnd = datetime.fromisoformat(render_iso)
    if rnd >= src: return "current"
    delta_hours = (rnd - src).total_seconds() / -3600.0
    return "stale" if delta_hours < 24 else "expired"

def pick_template(source: Path) -> str:
    # Infer dominant prefix from filename or content
    name = source.stem.upper()
    for prefix, tmpl in PREFIX_TEMPLATE.items():
        if prefix in name:
            return tmpl
    return DEFAULT_TEMPLATE

def render_one(source: Path, out_dir: Path, strict: bool=False) -> dict:
    post = frontmatter.load(source)
    md = markdown.Markdown(
        extensions=["fenced_code", "tables", "attr_list", "toc",
                    "pymdownx.superfences"],
        extension_configs={
            "pymdownx.superfences": {
                "custom_fences": [
                    {"name": "color-tokens", "class": "color-tokens",
                     "format": custom_fence_color_tokens},
                    {"name": "mermaid",      "class": "mermaid",
                     "format": custom_fence_mermaid},
                    {"name": "wireframe",    "class": "wireframe",
                     "format": custom_fence_wireframe},
                ]
            }
        }
    )
    body_html = md.convert(post.content)
    render_time = datetime.now(timezone.utc).isoformat(timespec="seconds")
    source_time = git_mtime(source)

    env = Environment(
        loader=FileSystemLoader(TEMPLATES),
        autoescape=select_autoescape(["html"]),
    )
    template = env.get_template(pick_template(source))
    html = template.render(
        title=post.metadata.get("title", source.stem),
        frontmatter=post.metadata,
        body=body_html,
        render={"time": render_time, "source": str(source.relative_to(REPO_ROOT)),
                "source_time": source_time, "generator_version": VERSION,
                "freshness": freshness(source_time, render_time),
                "template": pick_template(source)},
    )
    rel = source.relative_to(REPO_ROOT).with_suffix(".html")
    out_path = out_dir / rel
    out_path.parent.mkdir(parents=True, exist_ok=True)
    out_path.write_text(html, encoding="utf-8")
    return {"source": str(source.relative_to(REPO_ROOT)),
            "html": str(rel), "rendered_at": render_time}

# ... CLI wrapper, custom_fence_* helpers in _render/parser.py

Step 3 · GitHub Pages Workflow

~half day
Goal:
On every push to main, re-render all SoT/EPIC files and publish to GitHub Pages.
Inputs:
STEP-001 and STEP-002 complete; admin access to repo Settings → Pages.
Outputs:
Public URL serving rendered SoT/EPIC index.

Procedure

  1. Add docs/ to .gitignore (the gh-pages branch holds the publish target, not main).
  2. Create .github/workflows/render-sot.yml with the contents below.
  3. Implement an "all" mode in the generator: render-sot-html.py all --out docs/ walks SoT/ and epics/, renders each.
  4. Generator writes docs/index.html as a landing page listing all rendered files with their freshness pills.
  5. Push to a branch; verify the workflow runs green; merge.
  6. Settings → Pages → Source: gh-pages branch, / (root).
  7. Visit https://<owner>.github.io/<repo>/ and confirm the index renders.

Workflow file

# .github/workflows/render-sot.yml
name: Render SoT to GitHub Pages

on:
  push:
    branches: [main]
    paths:
      - 'SoT/**'
      - 'epics/**'
      - 'templates/**'
      - 'scripts/render-sot-html.py'
      - 'scripts/_render/**'
  workflow_dispatch: {}

permissions:
  contents: write

jobs:
  render:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }       # needed for git mtime per file

      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }

      - name: Install dependencies
        run: pip install -r scripts/requirements-render.txt

      - name: Render all
        run: python scripts/render-sot-html.py all --out docs/ --strict

      - name: Publish to gh-pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./docs
          publish_branch: gh-pages
          force_orphan: true
Acceptance Criteria
  • Workflow file passes actionlint (or equivalent) check
  • First run on the workflow_dispatch trigger completes green
  • A gh-pages branch exists with docs/ contents at its root
  • The Pages URL serves the rendered index and individual SoT files
  • Pushing a markdown change to an SoT file triggers re-render within 2 minutes

## Rendered Views

SoT and EPIC files are authored in markdown (canonical) and rendered to styled
HTML for human reading. Authors never edit HTML directly.

- **Generator**: `scripts/render-sot-html.py` (v0.1.0)
- **Templates**: `templates/` — one base + per-prefix specializations
- **Publish URL**: https://<owner>.github.io/<repo>/
- **Workflow**: `.github/workflows/render-sot.yml` (push to main triggers re-render)
- **Companion JSON**: every rendered HTML has a sibling `.json` for tooling

### Per-prefix templates

| Prefix | Template | What it adds |
|--------|----------|--------------|
| DES    | `sot-design.html.jinja`        | Color swatches, type specimens |
| ARC    | `sot-architecture.html.jinja`  | Mermaid diagrams |
| SCR/UJ | `sot-screen-flow.html.jinja`   | Wireframe boxes |
| EPIC   | `epic.html.jinja`              | Phase progress, session state |
| (default) | `sot-default.html.jinja`    | Generic entry rendering |

### Freshness states

Each rendered page shows a visible "Last rendered" block.
- ● **Current** — render is newer than source
- ● **Stale** — source modified after render, < 24h
- ● **Expired** — source modified > 24h before last render

CI re-renders on every push; expired state indicates a CI failure.

Contract

Every rendered HTML page MUST include a visible metadata block near the top with the following fields:

FieldTypeSourceRequired
last_renderedISO 8601 UTCGenerator invocation timeYes
source_pathString (repo-relative)Generator inputYes
source_last_modifiedISO 8601 UTCgit log -1 --format=%cI <file>Yes
generator_versionSemVer stringPinned in scriptYes
templateString (filename)Generator template selectionYes
freshnessEnum: current / stale / expiredComputedYes

Freshness computation

def freshness(source_iso: str, render_iso: str) -> str:
    src = datetime.fromisoformat(source_iso)
    rnd = datetime.fromisoformat(render_iso)
    if rnd >= src:

  • Current — green dot, label "Current"

  • Expired — red dot, label "Expired", with source mtime shown and CI status link if available

SPEC-002

Spec: Code-Fence Routing Conventions

Contract

Authors annotate markdown with fenced code blocks tagged with custom languages. The renderer recognizes these and produces rich HTML. Source markdown remains readable on GitHub (fences just appear as code blocks if not rendered).

Fence languageBody formatRendered asUsed by prefix
color-tokensname: #hex per lineSwatch gridDES
type-specimenYAML: family, weight, size, sampleTypography preview blockDES
mermaidMermaid syntaxDiagram (via Mermaid.js CDN)ARC
wireframeOne region per line, prefix [ ... ] for boxes, | for splitsBordered box layoutSCR, UJ
er-diagramSimple entity: field type, field type linesER diagramDBT (future)

Behavior

  • Unrecognized fence languages render as plain <pre><code> with the language as a class — same as today.
  • With --strict, unrecognized fences in source files under a specialized template are an error. Strict mode runs in CI.
  • All fence handlers receive the raw fence body and return HTML. They must escape user input to prevent XSS even though source files are trusted.

PIT-001

Pitfalls & Watch-outs

Read Before Starting

Drift discipline lives in CI

The visible freshness pill shows drift; it doesn't prevent it. The workflow must be reliable. If CI fails silently for a week, the pills go amber, then red, and readers stop trusting them. Treat workflow failures as alerts, not noise.

Strict mode is mandatory in CI

Without --strict, an unrecognized fence renders as a code block and authors never notice. Always run with --strict in the workflow. Local runs can omit it for fast iteration.

JSON schema is API once published

The first time the manifest goes live, the schema is locked. Adding fields is fine. Renaming or removing fields requires a version bump and a deprecation window. Sketch the schema in a draft PR before merging if uncertain.

Templates can become their own project

Per-prefix templates have unlimited surface area — dark mode, print stylesheets, interactive filtering, search. Resist. Ship the three specialized templates in scope and defer everything else. Re-evaluate after one month of use.

Mermaid CDN is a network dependency

Architecture diagrams depend on a CDN-loaded Mermaid.js. If offline reading matters, vendor Mermaid into templates/vendor/mermaid.min.js and reference it locally. If not, the CDN is fine.

Mid-rollout the docs site shows partial content

The first push to the gh-pages branch will list only the files rendered so far. Either render-all on the first publish, or accept a half-empty index for one workflow run.

The generator must read git for source mtime

File system mtime is unreliable in CI (checkout resets it). Always use git log for source_last_modified. The workflow includes fetch-depth: 0 for this reason.
