ASSESS-001

Executive Summary

Do Not Migrate

Converting SoT and EPIC files to HTML would yield modest human-readability gains while imposing a ~2.5–3× token-cost multiplier on every context load and a 6–10 working-day rewrite across 5 parsing scripts, 4 hooks, 38 skills, and every template in the repository.

The deeper risk is workflow fragility: large language models author and edit markdown more reliably than HTML. Unclosed tags, malformed table cells, and inconsistent nesting will compound into re-runs — a hidden operational tax that the bench tests above don’t capture.

Recommendation Keep markdown as the source of truth. Add a small generator (scripts/render-sot-html.py) that produces styled HTML views into a dist/ directory. You get the readability win in ~1 day with zero regression to tooling, skills, or token cost.

TOKEN-001

Token Consumption Analysis

High Cost

The same SoT entry — ID, status, body, related references — costs roughly 2.5–3× more tokens in HTML than in markdown. This applies to both input (loading SoT/EPIC files into context) and output (Claude writing a new entry).

Side-by-side: a single BR-001 entry

Markdown source~30 tokens
### BR-001: Free Plan Rate Limit
**Status**: Active
**Category**: Pricing

Free tier limited to 100 requests/day.

**Related**: API-045, UJ-012
Equivalent semantic HTML~80 tokens
<article id="BR-001" class="sot-entry">
  <h3>BR-001: Free Plan Rate Limit</h3>
  <dl>
    <dt>Status</dt><dd>Active</dd>
    <dt>Category</dt><dd>Pricing</dd>
  </dl>
  <p>Free tier limited to 100 requests/day.</p>
  <p>Related: <a href="#API-045">API-045</a>,
     <a href="#UJ-012">UJ-012</a></p>
</article>

Projected cost at repo scale

SurfaceMarkdown todayHTML projectedDelta
Load 2 SoT files into context ~2,000 tok ~5,500 tok +3,500
Load active EPIC ~500 tok ~1,250 tok +750
Write a new BR entry (output) ~100 tok ~250 tok +150
Per-session tax — — +4,000 tok typical
Hidden cost: output errors Markdown is forgiving (a missed asterisk is still readable). HTML is not (an unclosed <td> can break the whole table). Each malformed write becomes a re-run — invisible in synthetic benchmarks, but real in your monthly token bill.
How I arrived at ~2.5–3×

Tokenizers split on whitespace, punctuation, and common subwords. HTML wrapping introduces opening tags, closing tags, attribute names, attribute values, and quote characters — each contributing tokens that carry no semantic content. The multiplier is empirically stable for prose-heavy structured documents like SoT entries. Code-heavy or table-heavy content can land closer to 2×; sparse metadata-heavy entries can push 3.5×.

GAIN-001

What You’d Gain

Real Benefits
  • Native collapsibles — <details> elements (like the one above) reduce visual noise on long SoT files. Markdown has no first-class equivalent.
  • Anchor identity is structural — <h2 id="BR-001"> makes IDs first-class link targets without slug-mangling.
  • Status badges with color — visual scanability for status, severity, readiness (the pills in this document are a working demonstration).
  • Layout control — side-by-side panels, sticky navigation, grid layouts. Markdown can approximate these via embedded HTML but loses authoring ergonomics.
  • Browser-native rendering — no markdown preview extension needed; the file is the rendered view.
  • Sortable / filterable tables (with JavaScript) — not possible in markdown without a separate viewer.
Honest framing Most of these wins are achievable by rendering markdown to HTML on demand. You don’t have to change the source format to get the presentation benefits. See HYBRID-001.

LOSE-001

What You’d Lose

Real Costs
  • Diff readability — markdown diffs in PR reviews show content changes. HTML diffs show tag and content changes interleaved, often obscuring intent.
  • Hand-editability — quickly editing a SoT entry in VS Code is fluid in markdown (**bold**, | col |). In HTML you balance opening/closing tags every line.
  • GitHub native preview — GitHub renders markdown beautifully in-place. HTML files render as syntax-highlighted source (or stripped, depending on the path).
  • LLM authoring reliability — Claude (and most LLMs) produce far cleaner markdown than HTML. Every tag-mismatch is a fix-up round-trip.
  • Entire current toolchain — until rewritten, every script, hook, skill, and template in the repo stops working correctly.
  • Authoring ergonomics for humans — filling out a markdown template feels like writing. Filling out an HTML template feels like editing data entry.

TOOLING-001

Tooling Impact

Breaks Everything

The repository has no markdown AST parser. Every script uses regex against # heading syntax. The readiness scorer (scripts/_readiness/sot.py) anchors on the HEADING_DEF_RE pattern; scripts/validate-ids.sh regexes ^#{2,3} [A-Z]{2,5}-\d{2,3}; hooks scan ID patterns in source text. None of this survives a format change without rewrite.

ComponentParse strategyBreaks?Rewrite effort
scripts/validate-ids.sh Bash regex on ^## Yes ~2h
scripts/_readiness/sot.py Python regex on heading defs + entry body Yes ~6h (add BeautifulSoup, rewrite entry extraction)
scripts/_readiness/epic.py YAML frontmatter + body regex Partial ~3h (frontmatter unchanged; body regex needs HTML traversal)
scripts/check-stage-gate.sh Text grep Mostly ~1h
scripts/generate-id-pattern.sh Reads YAML, not markdown No 0
.claude/hooks/sot-update-trigger.sh Bash regex on ID pattern in diffs Partial ~1h (IDs still detectable in content)
.claude/hooks/context-validation.sh Text grep for version strings Partial ~1h
.claude/hooks/traceability-gate.sh Pattern-based Likely ~1h
Subtotal: tooling — ~15 hours

New runtime dependency required: beautifulsoup4 (or lxml) in Python scripts. Currently the only Python imports needed are yaml, json, and re.

SKILL-001

Skill Ecosystem Impact

Widespread

The repository ships 38 skills in .claude/skills/. Every skill that creates or reads a SoT entry currently emits markdown:

  • ghm-id-register — emits ### [ID]: [Title] headings with bold metadata fields.
  • ghm-sot-builder — its assets/sot-template.md generates markdown frontmatter, headings, and anchor links.
  • prd-v07-epic-scoping — structures EPIC entries with markdown tables and nested section headers.
  • Every prd-v0X-* skill (24 of them) that outputs IDs (BR-, UJ-, FEA-, etc.) embeds markdown formatting in its instructions.
  • ghm-gate-check, ghm-status-sync, ghm-harvest all read markdown structure to assess state.

Updating a skill is not just a search-and-replace — the SKILL.md files contain teaching examples that show Claude exactly what valid output looks like. Those examples must be rewritten and verified to produce parseable HTML on first attempt.

Estimate 38 skills × ~30–45 min each = ~20–28 hours for careful skill migration with at least one test invocation per skill to verify the new HTML output validates.

TPL-001

Template Impact

Twin Required

Every template needs an HTML twin: EPIC_TEMPLATE.md, .claude/skills/ghm-sot-builder/assets/sot-template.md, and any per-skill boilerplate. The templates themselves are simple to convert — the cost lives in the human ergonomics that follows.

JS-001

Do You Need JavaScript?

Depends on Goal
CapabilityJS required?
Static reading, anchors, navigation, collapsible sections No — semantic HTML + CSS is enough (this document proves it).
Sortable / filterable tables Yes — needs a small library or vanilla script.
Cross-document search Yes — needs an index + client-side search (e.g. Lunr.js).
Live readiness badges that update from status/readiness.json Yes — needs fetch + DOM update.
Server-side parsing (Python tools) No — use BeautifulSoup, not JS.

Honest read: Your stated goal — “easier to read for a human and more structured” — starts with no JS, but to actually surpass markdown’s readability (filtering through 100+ entries, cross-document search) you’ll want JS within a few weeks. At that point you’re effectively building a documentation site, which is fine — but it’s a different project.

PLAN-001

Implementation Plan (if you proceed)

6–10 Working Days
PhaseWorkEffort
1. Foundation Design HTML templates (SoT + EPIC). Add BeautifulSoup. Write a parser library that produces the same SoTEntry shape as today. Build regression tests against existing markdown fixtures. 1–2 days
2. Tooling rewrite Rewrite the 5 affected scripts (validate-ids, readiness/sot.py, readiness/epic.py, check-stage-gate, generate-id-pattern). Update the 4 hooks. Verify against fixtures. 2–3 days
3. Skill migration Update 38 skills. Rewrite teaching examples. Smoke-test each skill produces parseable HTML. 2–3 days
4. Content migration Convert 13 SoT files + all EPICs via a one-shot conversion script. Hand-fix edge cases. Update domain-profile.yaml extensions. 1–2 days
Total — 6–10 days + ongoing ~2.5× token cost
Risks during migration
  • Skills that pass an HTML smoke test can still emit broken HTML on novel inputs — you’ll discover failures in production.
  • Mid-migration the repo is in two formats; tooling has to handle both until phase 4 completes.
  • Existing PRs and branches reference markdown files. Merge conflicts are inevitable.

HYBRID-001

Hybrid Alternative

Recommended

Keep markdown as the canonical source. Add a small Python generator that renders SoT and EPIC files into styled HTML views.

Proposed shape

scripts/render-sot-html.py             # New generator
templates/sot-entry.html.jinja         # Per-entry template
templates/sot-file.html.jinja          # Per-file wrapper with TOC
dist/sot/SoT.BUSINESS_RULES.html       # Generated, gitignored or committed
dist/sot/index.html                    # Cross-file landing page

Why this dominates

  • Zero token-cost regression — Claude still reads markdown.
  • Zero skill changes — all 38 skills keep working unchanged.
  • Zero parser rewrites — existing regex tools still operate.
  • Full presentation control — you can iterate on HTML/CSS without touching content.
  • Effort: ~1 day for a basic Jinja2-based generator with the templates already designed here.
  • Optional next steps — serve dist/ via GitHub Pages; add live readiness badges by reading status/readiness.json; add client-side search with Lunr.js. Each is additive and reversible.

LITE-001

Lightweight Wins (no migration needed)

Quick
  • Customize VS Code markdown preview — ship a .vscode/markdown.css that styles SoT files with badges, status colors, anchor decorations. Cost: hours, not days.
  • Add an HTML report mode to readiness — readiness.py status --html emits a shareable dashboard. The data is already in status/readiness.json.
  • Render SoT to GitHub Pages — a single workflow step that runs the hybrid generator on push gives you a hosted, navigable site without local dependencies.
  • Improve markdown structure first — before changing format, audit whether your markdown is using its full vocabulary (definition lists, footnotes, GFM admonitions). You may discover the existing format is more capable than current usage.

REC-001

Final Recommendation

Three Tiers
  1. Best Hybrid (markdown source + generated HTML views). One day of work. Zero regression. Full readability win. Can evolve into a full doc site if you want. Reversible at any time.
  2. Acceptable Stay on markdown, improve presentation. Customize markdown preview styling. Add an HTML readiness report. Audit current markdown usage. Half a day to a day of work.
  3. Not recommended Full migration to HTML source. 6–10 days of rewrites. ~2.5× token cost forever. Authoring friction for humans and LLMs. The presentation wins are achievable without paying these costs.
If you still want to migrate Start with one SoT file as a pilot. Live with it for two weeks. Measure: token usage, edit time, LLM error rate when adding entries. The numbers above are estimates — a one-file pilot turns them into evidence before you commit to the full rewrite.
